# Homepage PM Copy Completed Roadmap Plan

## Executive Summary

Update `index.html` copy only. Do not redesign the page, replace the visual system, alter screenshots, or change app links.

The homepage should now sell the complete product story under the assumption that the substrate runtime roadmap is complete:

- build in the grid;
- branch the work;
- run the model;
- explain what changed;
- replay the decision;
- run one approved model across a coverage universe;
- use approved `=DATA()` and `=AI()` inside execution;
- replay decisions with the model state, data, provider evidence, AI method, memory, chronology, and time rules that applied then;
- use Audit to show what ran, what changed, which evidence was used, who approved it, and why the result was valid at that point in time.

The emotional convergence should be one operator story:

```text
The spreadsheet became a governed decision system.
```

Every major copy choice should collapse back into five verbs:

```text
build -> branch -> run -> explain -> replay
```

The implementation should be a surgical copy update. Keep the current clipped, finance-native voice. Avoid substrate internals in visible marketing copy.

## Current-Page Diagnosis

The current homepage has a good base structure:

- hero with core promise and proof lines;
- analyst/developer hook rail;
- screenshot-led product surface section;
- feature grid around assumptions, scenarios, sensitivity, Runs, Run Sets, and Audit;
- `=AI()` section;
- Run Later section;
- How It Works flow;
- Get Started plus YAP section;
- asof123 section;
- Reckoning Machines section.

The page is already PM-facing in tone. It is not overly engineering-heavy overall. The main gap is that it under-sells completed roadmap capabilities:

- Branch and lineage workflows are almost absent from the main narrative.
- "One model, many companies" appears in the hero but is not explained.
- `=DATA()` is presented as a formula interface, but not as governed mapping across providers, calendars, and entity identifiers.
- Replay is emotionally strong but not yet framed as stable governed reconstruction.
- Provider-backed replay is implied, not explained.
- Audit evidence is present, but the page does not yet name the PM review experience clearly enough.
- YAP and Model Memory are present, but should now be described as admitted decision context when approved.
- asof123 reads like architecture positioning; it should read as active runtime time authority.

The current page also risks slight message fragmentation. "Spreadsheet OS," "time machine," lineage, AI, institutional memory, and asof123 should not read like separate products. They should all support the same operator story: the spreadsheet is now a decision system that can be built, branched, run, explained, and replayed.

## Proposed Copy Architecture

Keep the page order mostly intact and introduce the complete product story through small changes:

1. Tighten the hero around the core operator story.
2. Add Branch/lineage and one-model-many-companies to the hero proof lines.
3. Revise the institutional decision section so it explains lineage, Branch, promote/reset, and replay in PM language.
4. Expand the product surface screenshot captions only where screenshots already imply the workflow.
5. Rename or refine the feature grid to cover lineage, coverage universe, approved data, replay, and Audit evidence.
6. Add a compact `=DATA()` panel or subsection near the existing AI section or features section.
7. Upgrade YAP and Model Memory copy in the existing YAP block.
8. Reframe the asof123 block as an active runtime capability.
9. Keep implementation nouns out of main visible copy; use product nouns and outcomes.

Use "governed" sparingly. It belongs in the hero and key proof sections, but repeating it every paragraph will make the page feel like compliance software. After the first few mentions, prefer approved, replayable, explainable, reviewable, versioned, historical, and institutional.

Recommended narrative bridge:

Spreadsheets used to be publishing artifacts: investment banking models, sell-side models, printed reports, static assumptions. Then the buyside started building live internal models for capital allocation, scenario analysis, and execution. Models forked. Assumptions drifted. AI entered the workflow. The spreadsheet became a living institutional decision system, but most software still treats it like a file. fin123 was built for what spreadsheets actually became.

Placement: keep this as one paragraph, either between the hero and the "operating system for governed institutional decisions" section or inside that section as a short narrative bridge before the four-panel grid. Do not expand it into a manifesto. Do not add Global Research Analyst Settlement or Spitzer history to the homepage.

Visual treatment: make the bridge read as one compact paragraph. It may use slightly larger text or a restrained standalone panel, but it should not become a giant manifesto section.

No wholesale layout change is needed.

Implementation priority:

1. Hero.
2. Narrative bridge.
3. Audit card rewrite.
4. Features section.
5. YAP/asof123 tightening.
6. Hook rail pruning.
7. Footer.

The first four changes should carry most of the repositioning. Do those cleanly before expanding lower-page copy.

## Risks

- Hero bloat: adding every completed capability to the hero could make the first viewport harder to scan.
- Jargon creep: Branch, replay, data mapping, and time authority can become too technical if not tied to PM outcomes.
- Message fragmentation: spreadsheet OS, time machine, lineage, AI, institutional memory, and asof123 can read like separate stories unless the copy keeps collapsing them into build, branch, run, explain, replay.
- "Governed" fatigue: the word is accurate internally but can sound compliance-heavy if repeated in every paragraph.
- Compliance-software tone: Audit and evidence language can become cold and defensive. Keep it tied to "why did the number move?" and "what did we know then?"
- Promote/reset drift: copy can accidentally sound like destructive Git behavior. Keep promote as reviewed reconciliation and reset as returning working analysis to a known-good state.
- Over-dense nav: adding new anchors for every capability may break the current compact nav.
- Screenshot mismatch: new copy may describe capabilities not visible in current screenshots. Keep claims in captions broad enough that screenshots still feel credible.
- Duplicate replay language: the page already says "time machine" several times. Strengthen the definition once, then avoid repeating it in every section.
- Developer hook drift: the hook rail has useful jokes, but too much developer framing can distract from PM-facing positioning. Prune repeated jokes before adding new hooks.
