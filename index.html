Three products in one stack

fin123 starts as a standalone spreadsheet system, extends into applications as a worksheet runtime, and scales into a hosted platform for governed delivery.

1. Standalone spreadsheet runtime

Use fin123 as a finance-grade spreadsheet and modeling system on your own machine. Build models, run scenarios, verify outputs, and ship deterministic artifacts with no server required.

The spreadsheet, rebuilt for finance

fin123 keeps the speed and mental model of spreadsheets, but gives financial models a real lifecycle, safer data handling, and the ability to run beyond the desktop.

Fast where Excel slows down

The grid and formula workflow remain familiar, but table computations run on a columnar engine designed for real workloads. Instead of recalculating entire spreadsheets repeatedly, fin123 compiles table plans and executes them efficiently against real datasets.

Versioned and auditable by default

Models follow a lifecycle: commit changes, build deterministic outputs, verify integrity, and release approved results. Every build records its inputs, parameters, and outputs so results can always be reproduced and audited.

Headless when you need scale

The same committed model can run across many tickers, scenarios, and parameter sets without someone driving a spreadsheet manually. Batch builds produce versioned outputs that can be compared, verified, and released as sets.

Connected to real market data

Vendor data and SQL sources synchronize into governed local parquet caches. Builds then run offline against those inputs, keeping results deterministic while still integrating with real financial data systems.

Deterministic by design

Determinism is enforced by architecture: sync is separated from build, builds run offline, and every output is fingerprinted and verifiable.

Offline builds

fin123 sync fetches vendor data and writes local parquet caches with provenance. fin123 build reads only local files and never touches a live database. Table computations compile as Polars LazyFrame plans and materialize at the end. When a plan does not specify a sort, fin123 applies a deterministic secondary sort so the same inputs produce byte-identical parquet outputs.

Verified outputs

Each build records workbook hash, input hashes, effective parameters (params_hash), scenario overlays (overlay_hash), and exports (export_hash). Verification fails if fingerprints drift. Releases are explicit: mark a build as approved and publish it as an immutable record. In production mode, releases can be gated on a verify pass.

Finance-safe joins and lookups

Silent row multiplication is unacceptable in finance. join_left defaults to many_to_one validation so right-side keys must be unique. lookup_scalar is exact-match and defaults to error on missing or duplicate keys. If a lookup is ambiguous or a join would multiply rows, the build fails with actionable diagnostics.

workflow 
fin123 sync   → local caches + provenance (sync_runs/)
fin123 commit → immutable snapshots (snapshots/)
fin123 build  → immutable builds (runs/) + outputs + hashes
fin123 verify → drift checks (params, overlays, exports, plugins)
fin123 release → approved outputs (releases/) + release sets

Open source — fin123-core

The core engine is open source under the Apache-2.0 license. It is a complete, standalone product — no database, no server, no account required. Install it, build models, and use the browser UI entirely on your own machine.

What's included (open source)

Workbook engine (scalar DAG + table LazyFrame evaluation), Lark LALR(1) formula parser, CellGraph with cross-sheet references and named ranges, local browser UI (FastAPI on localhost, canvas grid, keyboard-first), immutable versioning (snapshots, builds, artifacts), batch builds, diff engine, verify, GC, XLSX import with formula classification, and project templates.

Full lifecycle, no dependencies

Edit → Commit → Build → Verify. Every build is deterministic and reproducible from local files alone. The CLI and browser UI ship together. No external services needed.

Apache-2.0 license
Use, modify, and distribute freely. The core engine will always be open source.

Pod — enterprise and team layer (private)

fin123-pod is a proprietary add-on for teams that need shared state, automation, and governance. It extends fin123-core with database-backed infrastructure. Requires a license.

GitHub for worksheets

Pod provides a complete registry and governance path for worksheet artifacts. An analyst defines worksheet logic, fin123 compiles it into a deterministic artifact, and the artifact is pushed to a shared Postgres registry where it is versioned, reviewed, and promoted through approval stages. Released worksheet builds become the governed, distributable unit that downstream applications and internal tools consume directly.

Applications never reimplement spreadsheet logic. They embed released worksheet artifacts from the registry. When worksheet logic changes, teams review and approve the new version centrally, and every consumer picks up the approved release. The registry becomes the system of record for worksheet versions, creating a shared artifact layer between analysts, platform teams, and the applications they support.

Shared registry and runner

Pod extends fin123-core with the shared execution infrastructure teams need in production. SQL sync pulls Postgres tables into local parquet caches with full provenance. Bloomberg and plugin connectors bring vendor data into the same governed sync path. A Postgres-backed registry stores model versions, builds, and releases so teams operate against a single shared state.

The headless runner executes a specific (model_id, model_version_id) with parameter overrides and returns versioned outputs, turning centralized execution into an automatable backend capability. Multi-step workflows with replay let teams chain builds and promotions into repeatable operational sequences.

Production mode governance

In production mode, builds can be blocked on import parse errors, missing SQL schema guards, missing plugin pins, or an unreachable registry. Releases are gated on verify pass. Assertions with severity error hard-fail builds.

A familiar architecture — rediscovered

While building this system we realized something interesting.

The architecture we arrived at unintentionally recreates the structure of the original Lotus 1-2-3.

Lotus separated three ideas:

  • tabular data
  • worksheet definitions
  • the rendered spreadsheet

fin123 arrived at the same structure:

  ↓
WorksheetView
  ↓

  ↓

We didn't start by trying to copy Lotus.

We were trying to solve a practical problem: how to stop writing spreadsheet logic in application code.

But the result ended up looking very familiar.

Quick start (macOS — no Python required)

Download and extract fin123-core-0.4.0-macos-arm64.zip. Open Terminal in the extracted folder and run:

terminal 
# see bundled templates
./fin123 template list

# scaffold a demo project
./fin123 init demo --template demo_fin123

# build it
./fin123 build demo

# open the browser UI
./fin123 ui demo

Quick start (Windows — no Python required)

Download and extract fin123-core-0.4.0-windows-x86_64.zip. Open a terminal (Command Prompt or PowerShell) in the extracted folder and run:

terminal 
# see bundled templates
fin123.exe template list

# scaffold a demo project
fin123.exe init demo --template demo_fin123

# build it
fin123.exe build demo

# open the browser UI
fin123.exe ui demo

What the demo shows

The demo_fin123 template creates a project with tables, scalar formulas, PARAM() bindings, and assertions. After the build you can explore outputs, override parameters, diff two runs, and try the full commit → build → verify → release lifecycle.
