Why this matters

fin123 is built for teams that need Excel familiarity, but cannot afford Excel fragility. It keeps analyst speed and ease-of-use, while making runs reproducible, reviewable, and scalable.

Analyst-speed modeling, without changing the workflow

Analysts can author in a familiar grid and bring existing XLSX models in as first-class fin123 projects. The engine runs fast on real data using Polars query plans and lazy materialization, so “try another case” is cheap. You get the ergonomics analysts want — without inheriting spreadsheet chaos.

Scale the same model across cases and names

Professional teams do not run a model once. They run it across many tickers, scenarios, and parameter sets. fin123 treats this as a structured build process: the same committed model runs headlessly across a universe, producing versioned outputs you can diff and compare — not a manual spreadsheet ritual.

Reproducibility you can prove

Builds record the committed model version, effective parameters, scenario overlays, and input fingerprints. Verification checks that outputs match what was recorded. If anything drifts — model, inputs, parameters, overlays — verification fails. Reproducibility becomes a system property, not a social contract.

Separation that prevents silent drift

Model logic is versioned independently from vendor and market data. Bloomberg, Visible Alpha, internal warehouses, and alt-data land as SQL datasheets that sync into local parquet caches with provenance (query hash, output hash, rowcount). Workbook evaluation stays offline and deterministic — no live database queries during builds.

Parameters and scenarios are first-class, not hidden cell state

Parameters are explicit inputs, not invisible cell mutations. Scenario definitions are overlays on the same committed model, not forked workbooks. fin123 records effective params and scenario overlays per build so you can run “same model, different ticker” and know exactly what changed — and what did not.

Workflows produce artifacts — including AI, safely bounded

Scenario sweeps, batch runs, and post-processing are modeled as workflows that emit versioned artifacts with lineage. AI can be used as an artifact producer (notes, summaries, derived outputs), but it is never embedded in live cell evaluation. That boundary keeps core computation deterministic while still enabling modern automation.

Discipline applied to spreadsheets

Spreadsheet workflows blur editing, execution, and publication. fin123 forces boundaries: commit changes, build deterministic outputs, verify integrity, then release only what you approve. Nothing updates silently. Quant teams get repeatability; CTOs get auditability; analysts get speed and a workflow that feels familiar.

Run it

Keep the UI for authoring. Use the CLI for scale. Sync data once, then run offline.

terminal 
fin123 sync
fin123 commit
fin123 build --scenario base --set ticker=AAPL
fin123 verify-build <build_id>

Pod version (team mode)

fin123 runs locally-first, but teams can opt into a shared pod that centralizes models, data caches, custom plugins, and headless execution.

Share models, data, and plugins

The pod provides a shared registry for committed model versions, datasheet caches, and approved plugin bundles. Teams can reuse the same curated inputs and operators without copy-pasting files or rebuilding local state.

Headless automation for pipelines

Trigger runs from CI, schedulers, or internal tooling. The pod executes builds headlessly and returns versioned artifacts with provenance. This turns “run the model” into an automatable backend capability.

Determinism preserved

The same contracts apply in pod mode: offline evaluation, recorded hashes, and verification gates. Centralization improves collaboration without sacrificing reproducibility.

Local-first by default
You can adopt fin123 locally and add the pod later when you want centralized execution, shared caches, or team-wide governance.

System guarantees

These are the guarantees fin123 is designed to enforce — the contract teams can rely on.

Authoring that feels like modeling

Grid authoring without coupling logic to presentation. Formatting stays separate and never changes what the model computes.

Scenarios are first-class

Base, upside, downside are overlays on the same committed model. Table materialization can be reused when only scalar parameters change.

Parameters are explicit

Parameters are system inputs, not hidden state. Builds record effective parameters and a parameter fingerprint for reproducible runs.

SQL datasheets stay offline

Vendor and alt-data sync to local parquet caches with provenance. Builds never reach out to live databases.

Guardrails for lookups and joins

Fail fast on ambiguous joins, duplicate keys, and missing required data. Silent row multiplication is unacceptable in finance.

Versioned artifacts and verification

Outputs are versioned artifacts with lineage. Verification fails when outputs drift from the recorded model and inputs.

AI is an artifact producer, not a live dependency
AI steps can generate versioned artifacts, but core evaluation stays deterministic. AI is never embedded in live cell computation.

DB optional, pod optional

Run fully standalone on the filesystem. Enable the shared pod and registry when you want centralized execution, shared caches, and team governance.
