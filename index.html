fin123.dev

Works like Excel, but runs as a program.

A spreadsheet-driven research operating system.

I spent 15 years building models in Excel and 15 years building systems in code.

As an analyst, I wanted to:
  • update the model rapidly in a keyboard-driven, low overhead way
  • change drivers and run scenarios as real executions, not spreadsheet recalculations
  • run models with tens of thousands of rows without the engine choking
  • use LLMs inside the model - every step is visible, versioned, and reproducible
  • see what the model looked like when decisions were made
As a coder, I wanted to:
  • run spreadsheets headlessly instead of rebuilding the model in code
  • compose spreadsheet logic into workflows without bolting on python packages or using vba
  • embed models into html without duplicating logic across data, model, and UI
  • know every run was deterministic and auditable
  • run LLMs as explicit steps in versioned execution graphs - with full audit and policy control

Excel locks data, the calc engine, and the spreadsheet into one box.

fin123 breaks them apart.

The spreadsheet is no longer just a file. It is a program.

You can run it, version it, and ship it.

The gap between a spreadsheet and a system

Excel bundles the UI, the data, and the calculation engine into one file. That makes it easy to start and impossible to scale. The model works, but it cannot be versioned, reproduced, compared, or run headlessly. The analyst builds it. The engineer reimplements it. Nobody trusts the handoff.

fin123 replaces the calculation engine. The model is still built with formulas, tables, and assumptions -- but instead of recalculating cells inside a file, fin123 compiles the model and runs it as an executable program. Every run follows a defined execution plan. One model, one runtime, two perspectives. The same engine powers the browser demo and the full production system.

Bottom line: Two products, one system

fin123-core is the calculation engine I wanted as an analyst. fin123-pod is the application control I wanted as a coder. They are two ways of interacting with the same model and the same system.

analyst

fin123-core -- the analyst system

Flexibility, speed, and audit. Explore assumptions across surfaces, sweeps, and grids. Keep the states that matter. Compare outcomes. Return to prior work. Every build is deterministic, every result has a hash, and the full exploration path is preserved. No server required.

fin123-pod -- the coder system

Run fin123 models as a team. Compile once, run anywhere -- no reimplementation in JS or Python. Share, version, review, and release models with discipline. Use LLMs as governed steps in workflows, not as autonomous agents.

analyst

AI integrated natively, and safely

Describe a calculation. The system drafts it as a governed code module, validates it against policy, and applies it to your model. No coding required. Every add-in participates in the same deterministic, hashed builds as the rest of the model.

ai workbench 
fin123> ai draft addin "calculate compound annual growth rate from start and end values over n periods"
  Draft draft_0001 created (47 lines)

fin123> validate draft draft_0001
  Syntax       pass
  Imports      pass  (no forbidden modules)
  Eval/exec    pass  (no dynamic execution)
  Network      pass  (no outbound calls)
  Metadata     pass  (deterministic: true, version: 1)
  Entrypoint   pass  (register() found)

fin123> apply draft draft_0001
  Applied -> plugins/cagr.py

fin123> commit
  Build ID: 20260322_091455_run_1
  Outputs: 14 scalars, 3 tables   (was 12 scalars)

Draft, validate, apply

An LLM generates the code. An AST-based policy scanner decides whether it is safe: no forbidden imports, no eval/exec, no network access, no side effects. Validated drafts are applied to plugins/ and loaded into the runtime on the next build. Drafts are immutable -- revisions create new drafts, linked by lineage.

Why this matters

Analysts extend their own models without waiting on engineering. The generated code is pure and deterministic -- it computes a value, nothing else. It participates in the same hashed, verified builds as every other formula. The model stays auditable. The analyst stays in control.

Ships with fin123-core. Open source. Bring your own API key (Anthropic or OpenAI).

coder

Structured exploration

A sweep runs the same model multiple times while varying one parameter. Each row is a real execution, not a recalculated cell. This is where analyst exploration becomes something the system can execute repeatedly and deterministically.

sweep 
fin123> sweep revenue_growth 0.04 0.06 0.08 0.10 0.12

  revenue_growth  value_per_share  enterprise_value  Build ID
  0.04            901.22           1,944,882         20260320_run_1
  0.06            1,118.47         2,280,144         20260320_run_2
  0.08            1,402.63         2,805,251         20260320_run_3
  0.10            1,744.19         3,311,144         20260320_run_4
  0.12            2,163.51         4,010,021         20260320_run_5

In Excel, a Data Table recalculates cells. In fin123, a sweep runs the full model for each value. Every row has its own build, its own outputs, its own hash. A grid extends this to two parameters with matrix output. Every point is a state you can compare, keep, and return to.

This replaces Excel Data Tables with deterministic, auditable execution.

analyst + coder

From exploration to production

This is where the analyst workflow becomes a system. Three steps. No code rewrite between them. For the analyst, exploration becomes something you can trust and revisit. For the coder, a model becomes something you can run, embed, and ship.

execution model 
Spreadsheet (workbook.yaml)
        |
        |  fin123 build
        v
Build Outputs (scalars + tables)
        |
        |  fin123 worksheet compile
        v
Compiled Worksheet (immutable JSON, runs headlessly)
        |
        v
Applications / dashboards / batch jobs

Author

Analysts define the model in a workbook: parameters, formulas, table plans, assertions. The model can be a DCF valuation, a revenue forecast, a credit model, a scenario analysis -- anything that would otherwise live in a spreadsheet. As they explore, fin123 keeps a working memory of every state -- so the path to a result is never lost.

Compile

fin123 build compiles the model into an immutable worksheet with a deterministic content hash. The compiled worksheet contains the calculation graph, evaluated outputs, display formatting, flags, and a full audit trail.

Run

Applications run the compiled worksheet directly. The same compiled model runs in a dashboard, a batch job, a risk system, or an API. It runs headlessly, at scale, without anyone opening a spreadsheet. The model the analyst explored is the same model the engineer runs.

coder

Every number traces back to a model

Commits, diffs, releases, and reproducible runs. Built in, not bolted on. Analysts need every number to be explainable. Engineers need every run to be reproducible.

Every model in fin123 has commits, diffs, releases, and reproducible runs. When a number is produced, it traces back to a specific model version, a specific set of inputs, and a specific content hash.

model lifecycle 
fin123 commit  -> snapshot the model
fin123 build   -> compile + hash the outputs
fin123 verify  -> detect drift
fin123 release -> ship the compiled worksheet

Why this matters in finance

Regulators ask which model produced a number. Auditors ask whether the model changed between runs. Portfolio managers ask whether the forecast they approved is the same one running in production. fin123 answers all of these with a hash.

coder

fin123-pod for teams

Models become shared, versioned, and governed across the firm. Extends fin123-core with the infrastructure a team needs to run models in production.

Shared models, governed releases

Teams publish compiled models to a shared registry. Versions are reviewed and promoted through approval stages. Applications consume released versions -- the application never reimplements the model logic.

Headless execution at scale

Run released models headlessly with parameter overrides -- across batch jobs, dashboards, and APIs. Every run is reproducible and auditable, with a full provenance trail back to the model version that produced it.

LLMs as governed steps

LLMs participate in workflows as governed steps, not as autonomous agents. Inputs and outputs are validated. Runs are deterministic, inspectable, and persisted as artifacts -- not narrative text.

This is what I wanted as a coder: to use LLMs without giving up control of the system.
