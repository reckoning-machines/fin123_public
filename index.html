How fin123 works

fin123 separates model authoring, model execution, and model delivery.

execution model 
Spreadsheet (workbook.yaml)
        |
        |  fin123 build
        v
Build Outputs (scalars + tables)
        |
        |  fin123 worksheet compile
        v
Compiled Worksheet Artifact (immutable JSON)
        |
        v
Applications / dashboards / reports

Spreadsheet = model authoring

A YAML workbook with explicit parameters, formulas, table plans, and assertions. This is where the financial logic lives. Built deterministically from local files — no database, no network calls.

Worksheet = structured view of model output

A declarative YAML spec that projects build results into derived columns, display formats, sorts, and flags. This is what analysts define. Compiled at build time, not reimplemented in application code.

Compiled worksheet artifact = what applications consume

An immutable JSON file with full provenance and a deterministic content hash. Applications embed, serve, or verify the artifact directly. The calculations live in the worksheet — not your code.

One system, three products

1. Standalone spreadsheet runtime

Run finance-grade spreadsheets locally with deterministic builds, verification, and versioned artifacts. Fast grid UX, explicit parameters, headless execution. No server required.

Open source — Apache-2.0

2. Embeddable worksheet runtime

Compile models into worksheet artifacts that applications can render without reimplementing spreadsheet logic. Derived columns, flags, formatting, and provenance are resolved at compile time.

Open source — Apache-2.0

3. Hosted worksheet platform

Publish, approve, release, and serve compiled worksheets from a central system. Applications request released worksheet artifacts over an API. Worksheet lifecycle governance: draft → approved → released → archived.

Requires fin123-pod license

Get started

Install

Standalone binaries (no Python required) are available for macOS arm64 and Windows x86_64.

For source installs:

terminal 
git clone https://github.com/reckoning-machines/fin123-core.git
cd fin123-core
pip install -e ".[dev]"
fin123 doctor

Build your first model

The demo_fin123 template creates a project with tables, scalar formulas, PARAM() bindings, assertions, and a worksheet spec.

terminal 
# scaffold a demo project
fin123 init demo --template demo_fin123

# build it
fin123 build demo

# open the browser UI
fin123 ui demo

Why fin123

Financial systems repeatedly reimplement spreadsheet logic inside application code. fin123 moves that logic into deterministic worksheet artifacts.

Built like software

Commit → build → verify → release lifecycle. Versioning and audit are native.

Fast where Excel slows down

Table computations run on a columnar engine. fin123 compiles table plans and executes them against real datasets.

Headless execution

Run the same committed model across tickers, scenarios, and parameter sets without someone sitting in front of a spreadsheet.

Parameters and scenarios are first-class

Explicit system inputs and structured overlays, not hidden cell edits.

Finance-safe joins and lookups

join_left defaults to many_to_one validation so right-side keys must be unique. lookup_scalar is exact-match and errors on missing or duplicate keys. Silent row multiplication is rejected with actionable diagnostics.

Deterministic by design

Sync is separated from build. Builds run offline against local files. Every output is fingerprinted. fin123 verify detects post-build tampering. Identical inputs always produce identical outputs.

Bloomberg-connected

Vendor data can be synchronized into governed local caches for offline, reproducible builds. Requires fin123-pod.

lifecycle 
fin123 commit  -> immutable snapshots (snapshots/)
fin123 build   -> immutable builds (runs/) + outputs + hashes
fin123 verify  -> drift checks (params, overlays, exports)
fin123 release -> approved outputs (releases/) + release sets

Hosted platform — fin123-pod

fin123-pod is a proprietary add-on for teams that need shared state, automation, and governance. It extends fin123-core with database-backed infrastructure. Requires a license.

Worksheet registry and governance

Teams publish compiled worksheet artifacts to a shared Postgres registry. Artifacts are versioned, reviewed, and promoted through approval stages. Applications consume released artifacts from the registry instead of reimplementing spreadsheet logic.

Shared registry and runner

SQL sync pulls Postgres tables into local parquet caches with provenance. Bloomberg and plugin connectors bring vendor data into the same governed sync path. A Postgres-backed registry stores model versions, builds, and releases. The headless runner executes models by (model_id, model_version_id) with parameter overrides.

Production mode governance

Builds can be blocked on import parse errors, missing SQL schema guards, missing plugin pins, or an unreachable registry. Releases are gated on verify pass. Assertions with severity error hard-fail builds.
