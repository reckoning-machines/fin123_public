Commit → Build → Verify → Release

fin123 enforces a strict lifecycle. The goal is to make model changes explicit, builds repeatable, verification objective, and releases controlled.

Commit

Commit freezes model logic into an immutable version. It is the unit of review and collaboration: a stable reference that can be reproduced later. Data stays separate from the committed model.

Build

Build produces a deterministic artifact from a committed model. Builds are headless and write versioned outputs that can be compared, archived, and audited.

Verify

Verify confirms reproducibility and hash integrity. It checks that the build matches its recorded inputs and that the artifact bytes are consistent. A mismatch signals drift or tampering.

Release

Release promotes approved builds for production use. A release is an explicit pointer to a known-good build artifact. Nothing “updates silently.”

What fin123 is (and is not)
fin123 is deterministic model infrastructure. It applies software discipline to spreadsheet models—without trying to replicate Excel’s entire UI surface area, just the model-building core.

Key features

This is the system contract: what fin123 guarantees and how teams use it.

Deterministic execution

Builds are reproducible given the same committed model and the same local inputs. Inputs are fingerprinted and builds record enough provenance to prove what was used. Non-deterministic metadata exists only as metadata.

No PRINT button, by design

fin123 does not optimize for presentation artifacts. Outputs are build artifacts intended for review, verification, and downstream consumption. If you need a report, generate it from released data—don’t treat formatting as the system of record.

Two-graph computation model

fin123 separates scalar computation from table computation. Tables are composed as optimized query plans and then materialized; scalars resolve as a dependency graph with explicit references. This separation keeps evaluation predictable and reviewable.

Offline builds with optional pre-sync

External data access is isolated to a sync step that writes local caches. Builds never reach out to live databases. This prevents hidden dependencies and makes “same inputs → same outputs” enforceable.

Schema and lookup guardrails

Data contracts can be enforced: expected columns, stable keys, and join cardinality checks. The default posture is “fail fast” on ambiguous joins or missing required data—because silent row multiplication is unacceptable in finance.

Versioned artifacts and provenance

Builds produce immutable outputs and record the lineage: model version, input fingerprints, and the exact artifact identity. This supports diffing across runs, audit review, and reproducible research workflows.

Scenario and parameter sweeps

Running the same model across many parameter sets (e.g., many tickers) is a first-class workflow. Each build is independently identifiable, while still supporting batch-level grouping for reporting and release management.

Verify and release gates

Verification is a required step to establish integrity. Releases are explicit promotions of verified builds, enabling a clean separation between “research outputs” and “production-approved artifacts.”

Bounded storage with garbage collection

fin123 prevents unbounded disk growth by enforcing retention policies. The system protects recent and pinned artifacts while safely deleting old, unpinned history according to policy.

Extensible connectors and workflows

Data acquisition and auxiliary workflows are pluggable. Connectors produce cached datasheets; workflows produce versioned artifacts. AI can be used as an artifact producer, but it is not embedded in live cell evaluation.

DB optional: local-first by default

fin123 runs fully standalone on the filesystem. A database-backed registry and a headless runner can be enabled for team visibility and centralized execution, but they are not required for determinism.

Why this matters

Parallel builds across parameter sets

Analysts often need the same model executed across many entities and scenarios. fin123 treats this as a structured build process with traceable outputs—not a manual spreadsheet ritual.

Determinism and auditability

Results must be reproducible. fin123 provides an explicit lifecycle and objective verification so outputs can be trusted, reviewed, and reproduced without relying on a person’s machine state.

Separation of model and data

Model logic is versioned independently from vendor and market data. Data can be synced ahead of time, but evaluation remains offline. This keeps runs explainable and eliminates hidden dependencies.

Software discipline applied to spreadsheets

Spreadsheet workflows often blur editing, execution, and publication. fin123 forces boundaries: commit changes, build artifacts, verify integrity, and release only what you approve.
