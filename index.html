Why this exists

Applications repeatedly implement spreadsheet logic in HTML and JavaScript tables. fin123 replaces that with a deterministic worksheet runtime.

Every internal tool that displays tabular data eventually needs derived columns, formatting rules, flags, grouped headers, and audit context. Teams build these features ad hoc — one application at a time — with no shared runtime, no provenance, and no way to verify that the output is correct.

Define the worksheet as a YAML spec. Compile it against build outputs. Render the compiled artifact in a thin viewer. The runtime handles derived expressions, dependency ordering, error propagation, display formatting, and provenance — so the application code does not have to.

The architecture we arrived at unintentionally recreates the structure of the original Lotus 1-2-3.

Deterministic by design

Determinism is enforced by architecture: sync is separated from build, builds run offline, and every output is fingerprinted and verifiable.

Offline builds

fin123 sync fetches vendor data and writes local parquet caches with provenance. fin123 build reads only local files and never touches a live database. Table computations compile as Polars LazyFrame plans and materialize at the end. When a plan does not specify a sort, fin123 applies a deterministic secondary sort so the same inputs produce byte-identical parquet outputs.

Verified outputs

Each build records workbook hash, input hashes, effective parameters (params_hash), scenario overlays (overlay_hash), and exports (export_hash). Verification fails if fingerprints drift. Releases are explicit: mark a build as approved and publish it as an immutable record. In production mode, releases can be gated on a verify pass.

Finance-safe joins and lookups

Silent row multiplication is unacceptable in finance. join_left defaults to many_to_one validation so right-side keys must be unique. lookup_scalar is exact-match and defaults to error on missing or duplicate keys. If a lookup is ambiguous or a join would multiply rows, the build fails with actionable diagnostics.

workflow 
fin123 sync   → local caches + provenance (sync_runs/)
fin123 commit → immutable snapshots (snapshots/)
fin123 build  → immutable builds (runs/) + outputs + hashes
fin123 verify → drift checks (params, overlays, exports, plugins)
fin123 release → approved outputs (releases/) + release sets

Open source — fin123-core

The core engine is open source under the Apache-2.0 license. It is a complete, standalone product — no database, no server, no account required. Install it, build models, and use the browser UI entirely on your own machine.

What's included (open source)

Workbook engine (scalar DAG + table LazyFrame evaluation), Lark LALR(1) formula parser, CellGraph with cross-sheet references and named ranges, local browser UI (FastAPI on localhost, canvas grid, keyboard-first), immutable versioning (snapshots, builds, artifacts), batch builds, diff engine, verify, GC, XLSX import with formula classification, and project templates.

Full lifecycle, no dependencies

Edit → Commit → Build → Verify. Every build is deterministic and reproducible from local files alone. The CLI and browser UI ship together. No external services needed.

Apache-2.0 license
Use, modify, and distribute freely. The core engine will always be open source.

Pod — enterprise and team layer (private)

fin123-pod is a proprietary add-on for teams that need shared state, automation, and governance. It extends fin123-core with database-backed infrastructure. Requires a license.

What Pod adds

SQL sync (Postgres → local parquet caches with provenance), Bloomberg and plugin connectors, Postgres model registry (versions, builds, releases), headless runner service (POST /run), multi-step workflows with replay, release system with release sets, and production mode gates.

Registry + runner for centralized execution

Push model versions to a shared Postgres registry. The headless runner executes a specific (model_id, model_version_id) with parameter overrides and returns versioned outputs — turning "run the model" into an automatable backend capability.

Production mode governance

In production mode, builds can be blocked on import parse errors, missing SQL schema guards, missing plugin pins, or an unreachable registry. Releases are gated on verify pass. Assertions with severity error hard-fail builds.

Proprietary — contact us for access
Pod is closed source. It depends on fin123-core (open source) and adds enterprise capabilities. The core engine works fully standalone without Pod. Contact us for licensing.

Quick start (macOS — no Python required)

Download and extract fin123-core-0.4.0-macos-arm64.zip. Open Terminal in the extracted folder and run:

terminal 
# see bundled templates
./fin123 template list

# scaffold a demo project
./fin123 init demo --template demo_fin123

# build it
./fin123 build demo

# open the browser UI
./fin123 ui demo

Quick start (Windows — no Python required)

Download and extract fin123-core-0.4.0-windows-x86_64.zip. Open a terminal (Command Prompt or PowerShell) in the extracted folder and run:

terminal 
# see bundled templates
fin123.exe template list

# scaffold a demo project
fin123.exe init demo --template demo_fin123

# build it
fin123.exe build demo

# open the browser UI
fin123.exe ui demo

What the demo shows

The demo_fin123 template creates a project with tables, scalar formulas, PARAM() bindings, and assertions. After the build you can explore outputs, override parameters, diff two runs, and try the full commit → build → verify → release lifecycle.
