The build system for financial models

Excel familiarity. Deterministic builds. Audit-grade releases.

Author in a keyboard-first grid or run headless from the CLI. Compute fast on real data (Polars). Treat scenarios and parameters as first-class inputs for bulk updates. Sync vendor data via built-ins and SQL datasheets into local parquet caches. Build offline, verify drift objectively, and release only what you approve — locally, or through an optional team pod (registry + runner).

What’s new

Recent upgrades push fin123 from “deterministic spreadsheet engine” into a governed model lifecycle: versioned inputs, verifiable builds, and release-grade distribution — locally-first, with an optional team pod.

Pod mode: Registry + Runner

Optional Postgres registry for model versions, builds, releases, and release sets — plus a headless runner service (POST /run) to execute a specific (model_id, model_version_id) with parameter overrides. Local filesystem mode remains the default.

Release system + release sets

Releases are explicit approvals of immutable builds. Batch runs can be grouped into named release sets so “ship the universe build” becomes a single governed operation. In production mode, releases can be gated on a verify pass.

Workflows v1 + replay (idempotency)

Step-based workflows (sync/build/verify/batch/sweep) record per-step idempotency keys. Replay runs can generate a drift report when a workflow no longer produces identical keys under the same spec.

PARAM() bindings + parameter fingerprints

PARAM("name") proxy cells bind parameters directly to sheet locations. Builds record params_hash (effective params) and overlay_hash (scenario-only overrides) so you can prove exactly what changed between two builds.

Import hardening + review workflow

XLSX import now produces a versioned import report that classifies every formula (supported / unsupported function / parse error / external link / plugin formula). Unsupported formulas are never rewritten; review actions include mark-as-TODO and convert-to-value when safe.

Events, incidents, and project health

Runs and syncs emit structured NDJSON events. The UI surfaces failures as incident cards (build/verify/sync/import) and rolls them up into a single health status signal (ok / warn / error).

fin123 now supports a full supply chain: sync → commit → build → verify → release, with an optional pod for centralized execution and governance.

Why this matters

fin123 is for teams who need Excel’s design model but cannot accept spreadsheet fragility. It gives financial models a real lifecycle: commit changes, build deterministic outputs, verify drift, then release only what you approve. You keep analyst speed in a grid — and gain reproducibility, auditability, and scale across tickers, scenarios, and parameter sets in a governed model supply chain.

Spreadsheet UX, real lifecycle

Dark, dense, keyboard-first. The grid is fast, the UI stays out of your way, and common workflows are designed around navigation and batch edits. The difference is operational: your model has a lifecycle and produces versioned outputs you can audit and reproduce.

Stop saving files. Start shipping models.

Spreadsheet workflows blur editing, execution, and publication. fin123 enforces boundaries: commit changes, build deterministic outputs, verify integrity, then release only what you approve. Reproducibility becomes a system property, not a social contract.

Fast on real data

Table computations compile as Polars LazyFrame plans and materialize at the end. That means the engine optimizes work before it runs, and repeated “try another case” loops get cheaper. Sync once into local parquet caches; builds stay offline.

Scale one committed model across tickers and cases

Teams do not run a model once. They run it across many tickers, scenarios, and parameter sets. fin123 treats this as a structured build process: the same committed model runs headlessly across a universe, producing versioned outputs you can diff and compare. Batch runs stamp a shared build batch ID so outputs can be grouped and released as sets.

Parameters and scenarios are first-class

Parameters are explicit system inputs, not invisible cell mutations. Scenarios are overlays on the same committed model, not forked workbooks. fin123 records effective params and scenario overlays per build so you can say “same model, different ticker” and know exactly what changed — and what did not. Parameters can be bound to sheet cells using PARAM() proxy formulas, and builds record fingerprints for effective params (params_hash) and scenario overlays (overlay_hash).

Workflows produce artifacts — including AI, safely bounded

Scenario sweeps, batch runs, and post-processing are modeled as workflows that emit versioned artifacts with lineage. Workflows have step idempotency keys and optional replay reports to detect drift. AI can generate artifacts (notes, summaries, derived outputs), but it is never embedded in live cell evaluation. Core computation stays deterministic.

Common spreadsheet failures we eliminate

Broken links, “who edited this cell,” copy/paste drift, silent lookup fan-out, unreviewed changes, and “which file is the real one.” fin123 makes model state explicit, versioned, and verifiable — and surfaces issues as incidents (build/verify/sync/import) rather than hiding them in logs.

Excel compatibility envelope

fin123 keeps the spreadsheet mental model and supports XLSX import/export, but it is not trying to be Excel parity. The goal is deterministic modeling you can scale and audit.

What stays familiar

Grid authoring with a formula bar, multi-sheet workbooks, copy/paste TSV behavior, and XLSX import/export (values, formulas, and font color). Scalar formulas support Excel-like =expression syntax plus finance-oriented lookup semantics.

Formula engine + CellGraph

fin123 supports Excel-like expressions (+ - * / ^, comparisons, percent) and a growing function set. In the UI, the CellGraph evaluates cell formulas on-demand with memoization, cross-sheet references (Sheet!A1), named ranges for aggregates, and cycle detection. Errors render as #ERR! / #CIRC! instead of silently corrupting outputs.

XLSX export is deterministic

Export produces a clean XLSX for sharing: values, formulas, and font color (current scope). Exports are artifacts of a specific committed model + build inputs, and can be verified and diffed the same way as parquet outputs.

What fin123 intentionally does not do

No macros/VBA, no pivots, no charts, no conditional formatting, no volatile functions (NOW(), RAND()), no dynamic reference functions (INDIRECT(), OFFSET()), no spill/array formulas, no range expressions (A1:A10 — use named ranges), and NO PRINT BUTTON.

Import is best-effort, never silent

XLSX import preserves formulas as-is and generates a versioned import report that classifies every formula (supported, unsupported function, parse error, external link, plugin formula). Unsupported formulas are never rewritten; review actions include marking TODO and converting formulas to values when appropriate.

Built-ins: vendor data, governed ingestion

fin123 separates data acquisition from evaluation. Built-ins fetch vendor data during fin123 sync and write local parquet caches with provenance. Builds are offline and read only local files.

SQL datasheets → local parquet caches

Datasheets declare a connection and a query (inline or .sql file). Sync writes parquet caches and records provenance (query/spec hash, rowcount, output file hash) so you can reproduce and audit exactly what inputs were used.

Bloomberg / Visible Alpha built-ins

Built-ins are shipped integrations for third-party systems. They run during fin123 sync, write local parquet outputs, and record provenance. Bloomberg is sync-only and local-only (Terminal + blpapi), supporting BDP (reference) and BDH (historical) requests. Optional dependencies use safe detection: missing libraries are treated as skipped (with a warning), while service failures are fail with diagnostics.

Built-ins vs plugins

Built-ins are shipped connectors for vendor data and are enabled by project config and production-mode allowlists. Plugins are user-created code bundles (often AI-assisted) that add custom functions, workflows, and connectors — versioned and pinned so builds run with the exact operators you approved.

No live vendor calls during builds
sync is the only step allowed to touch external systems. build reads only local parquet/CSV caches so outputs are deterministic and verifiable.

Determinism contract

Determinism is enforced by design: sync is separated from build, builds run offline, and every build records the fingerprints needed to prove reproducibility.

Offline builds

fin123 sync fetches vendor data and writes local parquet caches with provenance. fin123 build reads only local files and never touches a live database.

Two-graph evaluation model

Table computations compile Polars LazyFrame plans and materialize at the end; scalar computations evaluate a named DAG with explicit dependencies. Lookups read from a run-local materialization cache, not ad-hoc queries.

Fingerprints: params, overlays, exports

Each build records workbook hash, input hashes, effective params (params_hash), scenario overlay (overlay_hash), plugin configuration (plugin_hash), and outputs (export_hash). Verification fails if recorded fingerprints drift.

Byte-stable exports

When a plan does not specify a sort, fin123 applies a deterministic secondary sort at export time so the same inputs produce byte-identical parquet outputs.

fin123 commit → immutable snapshots (snapshots/)
fin123 build  → immutable builds (runs/) + outputs + hashes
fin123 verify → drift checks (params, overlays, exports, plugins)
fin123 release → approved outputs (releases/) + release sets

Audit and governance

fin123 is designed for auditability: you can answer “what ran, with which inputs, under which assumptions, and who approved the output” without reconstructing history from filenames.

Every run is traceable

Builds record a workbook hash, input file hashes, effective parameters, scenario overlays, engine version, and (optionally) plugin pins. Verification fails if fingerprints drift. In production mode, verification and assertion policies can hard-fail builds.

Vendor data is versioned with provenance

Sync produces local parquet caches and a provenance record (query/spec hash, rowcount, output file hash, timestamps). Builds consume only these local versions, keeping evaluation offline and reproducible.

Release only what you approve

A release is an explicit act: mark a build as approved and publish it as an immutable record. Batch builds can be grouped into release sets so you can ship a universe run as one governed unit. In production mode, releases can be gated on a verify pass.

Events, incidents, and health

fin123 emits structured NDJSON events for runs and syncs and surfaces failures as incident cards: sync errors, build errors, verify failures, assertion failures, and import issues. Project health aggregates these into a single status signal (ok / warn / error).

Spreadsheet modeling as supply chain
Source (workbook) → dependencies (datasheets/built-ins/plugins) → build → verify → release. fin123 makes the chain explicit, versioned, and enforceable.

Finance-safe lookups and joins

Silent row multiplication is unacceptable in finance. fin123 defaults to deterministic, validated join and lookup semantics.

Deterministic joins with cardinality validation

join_left defaults to many_to_one validation so the right-side keys must be unique. Duplicate keys fail with sample duplicates; null keys are rejected under strict validation. Join key dtypes are checked up front to prevent silent mismatch.

VLOOKUP semantics with diagnostics

lookup_scalar is exact-match and defaults to error on missing or duplicate keys. Errors surface available columns and key samples for fast debugging. Declared primary keys can be enforced at materialization time.

Fail fast, not quietly
If a lookup is ambiguous or a join would multiply rows, the build should fail loudly with actionable diagnostics.

Run it

Keep the UI for authoring. Use the CLI for scale. Sync data once, then run offline.

terminal 
fin123 sync
fin123 commit
fin123 build --scenario base --set ticker=AAPL
fin123 verify-build <run_id>
fin123 release build <run_id>

# batch (universe)
fin123 batch build <dir> --params-file params.csv --scenario base
fin123 release set create --name spx_base --from-batch <build_batch_id> --require-verify-pass

# workflows (step-based)
fin123 workflow run build_and_verify <dir>
fin123 workflow run build_and_verify <dir> --replay wfr_0007

Pod version (team mode)

fin123 runs locally-first, but teams can opt into a shared pod: registry + runner for centralized execution, shared version tracking, and governance.

Optional registry (Postgres)

A Postgres-backed registry can store model versions, builds, releases, and release sets with idempotent upserts. Local filesystem mode remains the default; DB is opt-in for team visibility, automation, and shared audit trails.

Runner service for headless execution

A pod runner executes a specific (model_id, model_version_id) with parameter overrides via a simple API (POST /run) and returns versioned outputs. This turns “run the model” into an automatable backend capability for schedulers, pipelines, and services.

Prod mode gates

In production mode, builds can be blocked on import parse errors, missing SQL schema guards, missing plugin pins, or an unreachable registry. Releases can be gated on verify pass. Assertions with severity error can hard-fail builds.

DevOps for models, when you need it
The pod turns builds and releases into an automatable service while keeping the engine local-first and deterministic by default.

System guarantees

These are the guarantees fin123 is designed to enforce — the contract teams can rely on.

Analyst guarantees

Keyboard-first grid authoring, multi-sheet support, explicit parameters and scenarios, fast iteration on real data, and versioned builds you can diff. Imports never silently rewrite formulas; issues are surfaced in a report with a review workflow.

Operator guarantees

Offline builds, immutable snapshots and runs, bounded storage with GC + pinning, validated lookup/join semantics, structured events + incidents, and verification gates that fail when fingerprints drift.

AI is an artifact producer, not a live dependency
AI steps can generate versioned artifacts, but core evaluation stays deterministic. AI is never embedded in live cell computation.

DB optional, pod optional

Run fully standalone on the filesystem. Enable the shared pod and registry when you want centralized execution, shared version tracking, and team governance.

Plugins are user code, safely governed (AI-assisted)

Plugins are user-created bundles: custom scalar functions, table functions, workflows, and (optionally) custom connectors. AI can help generate plugin code and manifests, but plugins are versioned, pinned, and validated for engine compatibility. In production mode, missing plugin pins can block builds.
